# 羁绊判定与加成

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 羁绊优先、站位博弈、低门槛高上限

## Overview

羁绊判定与加成负责在对局中识别当前上阵单位的羁绊激活层级，并将对应效果稳定地写入战斗上下文。系统目标是让“构筑正确”在数值与体验上都可感知：玩家通过阵容组合获得明确收益，且收益规则可解释、可复盘、可平衡。

## Player Fantasy

玩家应感受到“我凑出的组合真的生效了”。当羁绊达成阈值，战斗表现会立即变化（更硬、更快、更痛或更稳定）；同一批武将因不同站位与羁绊组合能打出不同结果，体现构筑深度而非单卡面板碾压。

## Detailed Design

### Core Rules

1. 羁绊判定仅统计 `alive && deployed` 的友方单位，候场与死亡单位不计入。
2. 每条羁绊由 `match_tags`、`thresholds`、`effects_by_tier` 定义，数据源来自武将/羁绊数据模型。
3. 当单位上阵、下阵、死亡、复活（后续）或标签变化时触发重计算。
4. 羁绊层级取满足阈值的最大档位，未达最低阈值则为 `Inactive`。
5. 同条羁绊仅激活一个层级（最高层），不做同羁绊层级叠加。
6. 多条羁绊并存时按“先聚合再应用”流程执行，避免顺序抖动。
7. 加成写入运行时快照，不回写静态配置表。
8. 冲突型效果按 `priority` 解决；同优先级按效果类型固定顺序处理。
9. 每次重计算输出 `active_synergy_tiers` 与 `applied_modifiers` 供战斗消费。
10. 任何非法羁绊引用或参数越界不得中断对局，需降级并记录错误。

### States and Transitions

- `Idle`：非战斗相关阶段，仅保留最新判定快照。
- `Dirty`：检测到阵容变化，等待重计算。
- `Evaluating`：统计标签、计算层级、生成修正集。
- `Applied`：已回写到战斗上下文，等待下一次变更。

合法转移：`Idle -> Dirty -> Evaluating -> Applied -> Dirty`（循环）-> `Idle`。

### Interactions with Other Systems

- `武将/羁绊数据模型`：提供标签、阈值与效果定义；接收当前激活层级回写。
- `自动战斗结算`：读取羁绊修正后的战斗上下文（如攻防倍率、回能修正）。
- `九宫格布阵规则`：可读取邻接信息作为部分羁绊触发条件。
- `敌方AI目标选择`：可消费“嘲讽/后排保护”等羁绊状态作为目标偏好修正。
- `战斗HUD与商店UI`：显示羁绊激活计数、层级与下一档需求。
- `Playtest事件日志`：记录每次层级变化、触发来源与战斗结果关联。

## Formulas

### 羁绊计数

$$
count(s)=\sum_{u\in Team} \mathbf{1}[deployed(u)\land alive(u)\land tag_s\in tags(u)]
$$

### 层级判定

$$
tier(s)=\max\{k\mid threshold_{s,k}\le count(s)\}
$$

若集合为空，则 `tier(s)=0`（Inactive）。

### 加成聚合（同类乘区）

$$
Mul_{attr}=\prod_{m\in M_{attr}} (1+m)
$$

### 加成聚合（同类加区）

$$
Add_{attr}=\sum_{a\in A_{attr}} a
$$

### 最终属性修正

$$
Stat' = (Stat_{base}+Add_{attr})\cdot Mul_{attr}
$$

### 邻接触发类羁绊（可选）

$$
AdjBonus(u)=\eta\cdot Adj(u),\quad Adj(u)\le 4
$$

其中 `Adj(u)` 来自九宫格四方向邻接。

## Edge Cases

1. 同一单位拥有多个可匹配标签：允许同时贡献给多条羁绊计数。
2. 羁绊阈值配置无序：加载期排序；运行时使用排序后结果。
3. 阈值超过最大可上阵数：该档位标记不可达，不参与判定。
4. 战斗中单位死亡导致层级下降：下一 tick 生效，避免同 tick 连锁抖动。
5. 同帧上阵与下阵冲突：按事件时间戳与 `revision` 仅采纳最新快照。
6. 羁绊效果字段缺失：降级为无效效果并记录 `invalid_synergy_effect`。
7. 多羁绊同时改写同一属性：按乘区/加区规则聚合，禁止覆盖式写法。
8. 后台恢复后层级错乱：从最新阵容快照全量重算一次并覆盖缓存。
9. 调试热更参数导致不一致：以 `schema_version` 校验，不一致则拒绝热更。
10. 极端情况下修正过高：统一 clamp 到安全区间（由参数表定义）。

## Dependencies

### 上游依赖（Hard）

- `武将/羁绊数据模型`：提供标签、阈值、效果定义与实例状态。
- `自动战斗结算`：提供触发时机并消费修正结果。

### 下游依赖

1. `自动战斗结算`（Hard）：读取最终修正集参与伤害/能量计算。
2. `敌方AI目标选择`（Soft）：读取部分羁绊状态用于目标倾向。
3. `战斗HUD与商店UI`（Hard）：展示激活层级与下一档进度。
4. `Playtest事件日志`（Soft）：记录层级变化轨迹。

### 接口契约

- 输入最小字段：`match_id,wave,tick,team_instances,synergy_defs,layout_context`。
- 输出最小字段：`active_synergy_tiers,applied_modifiers,synergy_revision`。
- 同一 `match_id+wave+tick+team_revision` 必须给出确定性结果。

## Tuning Knobs

| Knob | 默认值 | 范围 | 说明 |
|---|---:|---:|---|
| `max_synergy_tiers` | 3 | 2–4 | 单羁绊层级上限 |
| `synergy_recalc_interval_tick` | 1 | 1–5 | 重算间隔（tick） |
| `modifier_clamp_min` | -0.8 | -1.0–0 | 负向修正下限 |
| `modifier_clamp_max` | 3.0 | 1.0–5.0 | 正向修正上限 |
| `adjacency_bonus_eta` | 0.03 | 0–0.1 | 邻接类羁绊系数 |
| `effect_priority_default` | 100 | 1–999 | 默认优先级 |
| `same_tick_apply_policy` | next_tick | enum | 同tick生效策略 |
| `synergy_debug_log_level` | normal | enum | 羁绊日志详细级别 |
| `max_active_synergies` | 8 | 4–12 | 同时激活羁绊上限 |
| `invalid_effect_policy` | degrade | degrade/error | 非法效果处理策略 |

## Visual/Audio Requirements

- 羁绊层级提升需触发统一事件 `synergy_tier_changed`。
- 首次激活与升档使用不同反馈强度（升档更显著）。
- 层级下降需有弱提示，避免玩家误解为系统异常。

## UI Requirements

- 战前/战中面板显示：当前激活羁绊、层级、计数/阈值、下一档差值。
- 提供羁绊来源追踪：点击羁绊可高亮贡献单位。
- 调试构建支持查看 `applied_modifiers` 明细与优先级排序结果。

## Acceptance Criteria

1. 1000 次同快照重算结果一致（层级与修正集一致）。
2. 羁绊计数严格基于 `alive && deployed`，候场/死亡不计入。
3. 多羁绊同属性修正聚合稳定，无顺序依赖抖动。
4. 非法配置可降级处理且不阻塞战斗流程。
5. 层级变化可被 UI 与日志完整观测，字段缺失率 `0`。
6. 与自动战斗集成后，战斗读到的 `synergy_revision` 与阵容版本一致。
7. 安卓目标机上单次羁绊重算耗时 `< 1ms`（MVP 规模）。
8. 邻接类羁绊在换位后可于预期时机生效（同 tick 或 next tick 按策略）。
9. 200 局模拟无羁绊死循环、无异常爆数、无负无穷。
10. 玩家可在 3 局内理解至少 1 条羁绊“激活->变强”的因果。
