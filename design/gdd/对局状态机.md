# 对局状态机

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 5分钟一局、低门槛高上限

## Overview

对局状态机是九宫军演单局流程的唯一节奏控制器，负责在5分钟目标内驱动每波的准备、招募、布阵、自动战斗与结算，并在合法状态间转换。它不处理战斗数值本身，只定义“何时允许做什么操作、何时结算、何时进入下一波/结束”，从而保证短局稳定、可恢复、可追踪。若缺少该系统，其他子系统（商店、战斗、波次、UI）将无法共享一致时序，导致规则冲突与体验割裂。

## Player Fantasy

玩家感受到的是“我在掌控整场战局节奏”：我知道当前阶段该做什么、还剩多少时间、下一步会发生什么。状态切换清晰且可预期，让玩家把认知资源用于构筑与站位决策，而不是与流程搏斗；同时切换节奏足够紧凑，形成“再来一局”的爽快循环。该系统属于基础设施型系统——理想状态是玩家几乎注意不到它，但会明显感受到顺滑、公平、节奏紧。

## Detailed Design

### Core Rules

1. 单局由 3 波组成：`Wave1(普通) → Wave2(精英) → Wave3(Boss)`。
2. 每波固定顺序：`PREPARE → SHOP → DEPLOY → BATTLE → RESOLVE`。
3. `SHOP` 仅允许执行招募、刷新、出售、锁定商店；禁止战斗相关操作。
4. `DEPLOY` 仅允许上阵/下阵/换位与阵型确认；不再允许刷新商店。
5. 进入 `BATTLE` 后，玩家输入冻结，直到战斗结束事件或超时事件。
6. `RESOLVE` 负责发放奖励、结算胜负、写入局内统计，并决定下一状态。
7. 任一状态若超时：执行该状态默认动作（如自动确认阵型）并前进，避免卡局。
8. 状态切换必须幂等：重复触发同一切换请求不应导致重复结算。
9. 对局提前结束条件：玩家生命值归零或到达最终波次并完成结算。
10. 任何系统只能通过状态机公开事件请求切换，不允许直接改写全局状态。

### States and Transitions

| State | Description | Valid Transition |
| --- | --- | --- |
| `BOOT` | 初始化对局上下文、种子、波次索引 | `WAVE_PREPARE` |
| `WAVE_PREPARE` | 装载当前波配置、重置临时标记 | `SHOP` |
| `SHOP` | 招募与经济决策窗口 | `DEPLOY`（`confirm`/`timeout`） |
| `DEPLOY` | 九宫格布阵窗口 | `BATTLE`（`confirm`/`timeout`） |
| `BATTLE` | 自动战斗运行中 | `RESOLVE`（`battle_finished`/`battle_timeout`） |
| `RESOLVE` | 奖励、伤害、任务、日志结算 | `WAVE_PREPARE`/`GAME_WIN`/`GAME_OVER` |
| `GAME_WIN` | 胜利结算与结果展示 | 终态 |
| `GAME_OVER` | 失败结算与快速重开入口 | 终态 |

### Interactions with Other Systems

- `招募商店与刷新经济`：输入 `state_enter(SHOP)`；输出 `shop_ops_closed` 事件供状态机切换。
- `九宫格布阵规则`：输入 `state_enter(DEPLOY)`；输出 `formation_locked`。
- `自动战斗结算`：输入 `state_enter(BATTLE)` 和阵容快照；输出 `battle_finished(result, duration)`。
- `波次生成（普通/精英/Boss）`：在 `WAVE_PREPARE` 拉取当前波敌人配置。
- `局内资源与奖励流`：在 `RESOLVE` 写入奖励与消耗。
- `战斗HUD与商店UI`：状态机广播 `state_changed`，UI据此切换面板与按钮可用性。
- `Playtest事件日志`：每次状态进出都写日志（含时间戳、波次、关键指标）。

## Formulas

### 变量定义

- `T_match`：单局目标时长（秒），默认 `300`
- `N_wave`：总波次，默认 `3`
- `T_shop`, `T_deploy`, `T_battle`, `T_resolve`：对应阶段时长
- `HP_p`：玩家生命值
- `D_w`：第 `w` 波失败惩罚伤害
- `R_w`：第 `w` 波基础奖励金币
- `B_w`：第 `w` 波胜利奖励倍率（普通=`1.0`，精英=`1.2`，Boss=`1.5`）

### 时长预算

$$
T_{wave} = \frac{T_{match}}{N_{wave}} = 100s
$$

$$
T_{shop}=25,\;T_{deploy}=20,\;T_{battle}=45,\;T_{resolve}=10
$$

$$
T_{shop}+T_{deploy}+T_{battle}+T_{resolve}=100
$$

超时前进规则：若任一阶段实际时长 $t_i > T_i$，触发默认动作并切换下一状态。

### 失败伤害（`RESOLVE` 执行）

$$
D_w = \{2,3,5\}\;\text{for}\;w=\{1,2,3\}
$$

$$
HP_p' = \max(0, HP_p - D_w)
$$

### 波次奖励（`RESOLVE` 调用奖励流）

$$
GoldGain_w = \lfloor R_w \cdot B_w \rfloor,\;R_w=\{4,5,6\}
$$

默认奖励为：Wave1=`4`，Wave2=`6`，Wave3=`9`（取整后）。

### 终局判定

$$
\\text{if } HP_p'=0 \\Rightarrow GAME\\_OVER
$$

$$
\\text{if } w=N_{wave}\\;\\&\\;result=win \\Rightarrow GAME\\_WIN
$$

否则进入下一波 `WAVE_PREPARE`。

## Edge Cases

1. 状态重复触发：同一帧内收到重复 `confirm`，仅首次生效，其余忽略（幂等键：`match_id + wave + state + tick`）。
2. 战斗超时：`BATTLE` 超过上限仍未结束，强制判定为“玩家失败（或平局按失败处理）”并进入 `RESOLVE`。
3. 空阵容开战：`DEPLOY` 结束时若上阵数为 `0`，允许进入战斗但直接触发失败分支（避免阻塞）。
4. 非法状态跳转请求：例如 `SHOP -> RESOLVE`，统一拒绝并记录错误日志，不改变当前状态。
5. 波次配置缺失：`WAVE_PREPARE` 未拉到配置时，回退到默认波次模板并打 `severity=high` 日志。
6. 玩家在状态切换边界操作：切换瞬间的买卖/换位请求进入队列，按“新状态可执行性”二次校验。
7. 生命值越界：所有写入统一 clamp 到 `[0, HP_max]`。
8. 应用切后台/恢复：恢复后重放最后一次已确认状态，不重复发奖励；若处于 `BATTLE`，根据策略直接判定超时并进入 `RESOLVE`。
9. Boss 波结算与死亡同时发生：以“先生命归零判败”优先，避免同局同时触发 `GAME_WIN` 与 `GAME_OVER`。
10. 快速重开连点：`GAME_OVER/GAME_WIN` 仅允许一次重开事务，后续点击忽略直到新局 `BOOT` 完成。

## Dependencies

### 上游依赖（Hard）

- 无硬上游（该系统为基础系统）。

### 下游依赖（本系统输出给谁）

1. `九宫格布阵规则`（Hard）
	- 输入接口：`on_state_enter(DEPLOY)`, `on_state_exit(DEPLOY)`
	- 输出期望：`formation_locked` 事件供状态机消费
2. `招募商店与刷新经济`（Hard）
	- 输入接口：`on_state_enter(SHOP)`, `shop_time_remaining`
	- 输出期望：`shop_ops_closed`
3. `自动战斗结算`（Hard）
	- 输入接口：`on_state_enter(BATTLE, battle_context)`
	- 输出期望：`battle_finished(result, duration, casualties)`
4. `波次生成（普通/精英/Boss）`（Hard）
	- 输入接口：`on_state_enter(WAVE_PREPARE, wave_index)`
	- 输出期望：`wave_payload_ready`
5. `局内资源与奖励流`（Hard）
	- 输入接口：`on_state_enter(RESOLVE, result_context)`
	- 输出期望：`resolve_applied`
6. `存档/读档（基础）`（Soft→后期Hard）
	- 输入接口：`snapshot_request`（关键状态点）
	- 输出期望：`snapshot_saved`
7. `战斗HUD与商店UI`（Hard）
	- 输入接口：`state_changed(prev, curr, wave, timer)`
	- 输出期望：纯表现层，无反向控制权
8. `Playtest事件日志`（Soft）
	- 输入接口：`state_enter/exit`, `transition_reject`
	- 输出期望：埋点写入成功/失败回调（不阻塞主流程）

### 接口约束

- 状态机是流程主控，其他系统只能“请求”切换，不能“强制”切换。
- 统一事件总线字段：`match_id`, `wave`, `state`, `timestamp_ms`, `source_system`。
- 任何依赖失败（除战斗核心）默认降级处理，不得阻塞状态前进。

## Tuning Knobs

| Knob | 默认值 | 安全范围 | 过低风险 | 过高风险 |
|---|---:|---:|---|---|
| `shop_duration_sec` | 25 | 15–35 | 决策仓促，构筑随机性过强 | 拖慢节奏，超出5分钟目标 |
| `deploy_duration_sec` | 20 | 10–30 | 站位价值下降 | 对局节奏拖沓 |
| `battle_timeout_sec` | 45 | 30–60 | 频繁超时误判 | 长时间等待、反馈延迟 |
| `resolve_duration_sec` | 10 | 5–15 | 信息读不完 | 节奏中断感强 |
| `wave_count` | 3 | 3–5 | 内容层次不足 | 超时风险增大 |
| `wave_damage_table` | [2,3,5] | 每项1–8 | 失败惩罚无感 | 挫败过强、重开流失 |
| `wave_reward_base` | [4,5,6] | 每项2–12 | 经济干瘪，流派难成型 | 经济膨胀、平衡崩坏 |
| `wave_reward_multiplier` | [1.0,1.2,1.5] | 1.0–2.0 | 波次区分不明显 | Boss收益失衡 |
| `auto_confirm_on_timeout` | true | bool | 可能卡局（若false） | 减少玩家控制感（若总是强制） |
| `transition_retry_limit` | 1 | 0–3 | 瞬时失败容错不足 | 重复结算风险上升 |

并行关系说明：`shop_duration_sec + deploy_duration_sec + battle_timeout_sec + resolve_duration_sec` 决定单波总时长，必须受 `T_match / wave_count` 约束。

## Visual/Audio Requirements

- 状态切换需有统一转场反馈：轻量遮罩 + 阶段标题（`<300ms`）。
- `SHOP` 倒计时进入最后 `5` 秒触发视觉闪烁与节奏音提示。
- `BATTLE` 开始/结束有明确音效边界，避免玩家误判阶段。
- `RESOLVE` 奖励发放采用“先结果后资源变化”顺序动画。
- 所有反馈优先低成本实现（像素风 + 最小粒子），确保安卓中低端稳定。

## UI Requirements

- 顶部固定状态条：当前波次、当前状态、倒计时、玩家生命值。
- `SHOP` 仅开放商店操作按钮；`DEPLOY` 仅开放布阵按钮；按钮禁用态必须可见。
- 状态切换时清空不合法交互焦点，避免“上一状态按钮残留可点”。
- 终态界面提供“快速重开”主按钮（1次点击生效，防抖）。
- 提供开发调试面板（仅调试构建）：手动跳状态、注入超时、打印当前状态。

## Acceptance Criteria

1. 流程完整性：单局必经 3 波，每波状态顺序严格为 `WAVE_PREPARE→SHOP→DEPLOY→BATTLE→RESOLVE`。
2. 状态合法性：自动化测试中所有非法跳转请求被拒绝且不改变当前状态。
3. 超时容错：`SHOP/DEPLOY/BATTLE` 超时均能自动前进，不出现卡死。
4. 幂等安全：重复 `confirm`/重复结算事件不会导致奖励重复发放。
5. 终局唯一性：每局只能到达一个终态（`GAME_WIN` 或 `GAME_OVER`）。
6. 时长目标：标准参数下，100 局模拟平均局时在 $300s \pm 30s$。
7. 稳定性：连续 200 局无人为干预模拟无崩溃、无死循环。
8. 跨系统联动：进入 `SHOP/DEPLOY/BATTLE` 时，UI 与对应子系统都收到一致 `state_changed` 事件。
9. 日志完整：每次状态进出均记录 `match_id/wave/state/timestamp`。
10. 移动端恢复：切后台再恢复后，不出现状态回退与重复发奖。
