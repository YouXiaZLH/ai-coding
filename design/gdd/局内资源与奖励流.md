# 局内资源与奖励流

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 5分钟一局、低门槛高上限、羁绊优先

## Overview

局内资源与奖励流负责在每波 `RESOLVE` 阶段统一结算金币、生命、胜负奖励与惩罚，并把结果稳定回写到对局上下文。系统目标是建立可预测又有抉择空间的资源循环：玩家通过正确构筑与运营获得下一波优势，同时失败也能快速复盘并重开。

## Player Fantasy

玩家应感受到“每一波结果都会改变我的下一步选择”。赢了有可感知的经济跃迁，输了有明确代价但不至于直接崩盘；奖励结构支持“保连胜滚雪球”与“逆风找翻盘”两类玩法。

## Detailed Design

### Core Rules

1. 仅在状态机进入 `RESOLVE` 时执行奖励流；其他阶段拒绝资源写入。
2. 结算输入最小包含：`battle_result`、`wave_index`、`wave_type`、`pre_gold`、`pre_hp`。
3. 胜利时发放基础奖励 + 波次倍率奖励；失败时扣除生命并仅发放保底收益（可配置）。
4. 资源写回顺序固定：`HP -> Gold -> Streak -> Flags -> Logs`，避免并发抖动。
5. 同一波次只允许一次有效结算，重复调用必须幂等返回同结果。
6. 结算后输出 `resolve_applied` 事件供状态机决定下一状态。
7. 金币与生命均做边界裁剪：金币 `[0,gold_cap]`，生命 `[0,hp_cap]`。
8. Boss 波额外结算项可独立配置（如额外金币、任务计数）。
9. 资源流不直接操作商店内容，仅更新可被商店消费的资源字段。
10. 所有结算写入日志：来源、计算过程、增减量、最终值、事务ID。

### States and Transitions

- `Idle`：非结算阶段，等待 `RESOLVE` 触发。
- `Collecting`：收集战斗与波次上下文。
- `Calculating`：执行奖励/惩罚公式并生成变更集。
- `Applied`：写回资源并广播 `resolve_applied`。

合法转移：`Idle -> Collecting -> Calculating -> Applied -> Idle`。

### Interactions with Other Systems

- `对局状态机`：主调方；传入 `result_context` 并消费 `resolve_applied`。
- `自动战斗结算`：提供胜负、伤亡、时长与关键统计。
- `波次生成（普通/精英/Boss）`：提供波次类型与难度参数。
- `招募商店与刷新经济`：消费结算后金币与连胜状态用于下波运营。
- `轻度局外解锁`：消费终局局外点数增量 `meta_point_delta` 作为解锁货币来源。
- `存档/读档（基础）`：保存关键结算点资源快照（后续阶段）。
- `战斗HUD与商店UI`：展示本波奖励明细与生命变化。
- `Playtest事件日志`：记录资源曲线与回合收益结构。

## Formulas

### 胜利金币

$$
GoldGain_w = \lfloor R_w \cdot B_w \rfloor + StreakBonus
$$

默认：$R_w=\{4,5,6\}$，$B_w=\{1.0,1.2,1.5\}$。

### 失败生命惩罚

$$
HP' = \max(0, HP - D_w)
$$

默认：$D_w=\{2,3,5\}$。

### 失败金币保底

$$
GoldGain_{lose} = \lfloor R_w \cdot LoseMul \rfloor
$$

MVP 默认 `LoseMul=0.5`。

### 连胜奖励

$$
StreakBonus = \min(S_{cap},\max(0, win\_streak-1)\cdot S_{step})
$$

默认：`S_step=1`，`S_cap=3`。

### 回合末总金币

$$
Gold_{next} = clamp(Gold_{pre} + GoldGain + Interest,\ 0,\ gold\_cap)
$$

其中 `Interest` 由商店经济规则计算并注入。

### 终局判定关联

$$
if\ HP' = 0 \Rightarrow GAME\_OVER
$$

$$
if\ wave=3 \land result=win \Rightarrow GAME\_WIN
$$

## Edge Cases

1. 同一波重复收到 `battle_finished`：按 `match_id+wave` 幂等键忽略重复结算。
2. 胜负结果缺失：标记 `resolve_invalid_input`，按失败兜底并报警。
3. 生命扣减后为负值：统一 clamp 到 `0`。
4. 金币计算溢出或负值：统一 clamp 到 `[0,gold_cap]`。
5. Boss 波奖励配置缺失：回退到精英波配置并记录高优先级日志。
6. 切后台恢复重复进入 `RESOLVE`：返回上次事务结果，不重复加减。
7. 连胜在失败后未清零：强制 `win_streak=0` 并记录修复事件。
8. 失败保底过高导致逆风滚雪球：由参数上限控制并在日志监控。
9. 状态切换边界资源写入冲突：仅接受 `RESOLVE` 来源写操作。
10. HUD 拉取到旧资源：以 `resource_revision` 为准触发二次刷新。

## Dependencies

### 上游依赖（Hard）

- `对局状态机`：提供 `RESOLVE` 生命周期与终局切换。
- `自动战斗结算`：提供本波胜负与战斗摘要。
- `波次生成（普通/精英/Boss）`：提供波次类型与参数。

### 下游依赖

1. `招募商店与刷新经济`（Hard）：消费金币与连胜状态。
2. `存档/读档（基础）`（Soft→后续Hard）：保存资源快照。
3. `战斗HUD与商店UI`（Hard）：显示奖励和惩罚反馈。
4. `Playtest事件日志`（Soft）：消费资源变化轨迹。

### 接口契约

- 输入最小字段：`match_id,wave,result,wave_type,pre_gold,pre_hp,battle_summary`。
- 输出最小字段：`post_gold,post_hp,gold_delta,hp_delta,meta_point_delta,resource_revision`。
- 同一波次结算必须幂等：重复请求返回同 `resource_revision`。

## Tuning Knobs

| Knob | 默认值 | 范围 | 说明 |
|---|---:|---:|---|
| `reward_base_wave` | [4,5,6] | 每项2–12 | 波次基础奖励 |
| `reward_mul_wave` | [1.0,1.2,1.5] | 1.0–2.0 | 波次奖励倍率 |
| `lose_hp_penalty_wave` | [2,3,5] | 每项1–8 | 失败扣血 |
| `lose_gold_mul` | 0.5 | 0–1.0 | 失败保底倍率 |
| `streak_step` | 1 | 0–3 | 连胜奖励步进 |
| `streak_cap` | 3 | 0–5 | 连胜奖励上限 |
| `gold_cap` | 99 | 30–200 | 金币上限 |
| `hp_cap` | 20 | 10–40 | 生命上限 |
| `boss_extra_reward` | 0 | 0–10 | Boss额外金币 |
| `resolve_retry_limit` | 1 | 0–3 | 结算重试上限 |

## Visual/Audio Requirements

- 结算展示顺序固定：胜负标题 -> 生命变化 -> 金币变化 -> 连胜变化。
- 奖励增长使用正反馈色与短音效；扣血使用负反馈色与轻震动（移动端可选）。
- Boss 波胜利结算可增加一次强化反馈（不超过300ms）。

## UI Requirements

- `RESOLVE` 面板显示：本波奖励明细、失败惩罚、当前总资源。
- 展示“下一波可用金币预测”（含商店利息估算）。
- 调试面板可查看 `resource_revision` 与结算公式中间值。

## Acceptance Criteria

1. 每波 `RESOLVE` 仅触发一次有效资源写回，无重复发奖。
2. 与状态机联动正确：结算后可稳定进入下一波或终局。
3. 金币与生命永不越界，资源边界裁剪正确。
4. 同输入重放 1000 次结算结果一致。
5. 胜负奖励与惩罚曲线与设计表一致，无配置漂移。
6. 下游商店读取到的金币与连胜状态与结算结果一致。
7. 结算流程平均耗时 `< 0.5ms`（MVP 规模）。
8. 失败兜底存在但不破坏挑战性（可通过对局数据验证）。
9. 日志可完整追溯每波资源变化来源。
10. 玩家能在 3 局内理解“赢多赚、输掉血、下波可调整”的闭环。
