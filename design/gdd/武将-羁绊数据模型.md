# 武将/羁绊数据模型

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 羁绊优先、低门槛高上限

## Overview

武将/羁绊数据模型定义“谁是单位、单位拥有哪些可计算属性、羁绊如何被判定并输出加成”，是自动战斗、商店、布阵与 UI 共享的唯一数据真源。该模型采用“静态定义 + 局内实例状态”双层结构：静态层描述武将与羁绊规则，实例层记录当前对局中的生命、能量、位置、星级与状态标记。系统目标是可读、可验证、可扩展，确保在 5 分钟快节奏内仍能稳定支撑构筑表达与平衡迭代。

## Player Fantasy

玩家应感受到“我在用清晰规则构筑强阵容”：每个武将的身份、定位、羁绊归属都直观可理解；当阵容达成阈值时，反馈明确且可预期。该系统服务于“羁绊优先”支柱，让玩家通过组合与取舍获得优势，而非依赖黑盒数值。对玩家而言，数据模型不应显形为复杂配置，而应表现为“我看得懂、算得出、能复盘”的策略语言。

## Detailed Design

### Core Rules

1. 数据模型分两层：`StaticConfig`（静态配置）与 `RuntimeInstance`（局内实例）。
2. 静态配置在对局 `BOOT` 时加载并只读，不允许局内修改。
3. 每个武将配置必须包含：`general_id`、`name_key`、`faction_tags`、`role_tags`、`cost`、`base_stats`、`skill_id`。
4. 每个羁绊配置必须包含：`synergy_id`、`match_tags`、`thresholds`、`effects_by_tier`、`priority`。
5. 局内实例最小字段：`instance_id`、`general_id`、`star_level`、`level`、`current_hp`、`current_energy`、`board_pos`、`alive`。
6. 任何战斗计算只能读取实例快照，不直接读写静态配置。
7. 羁绊判定按“上场有效单位”统计，不包含候场位。
8. 当单位/站位/存活发生变化时，触发羁绊重计算事件。
9. 数据校验分两阶段：加载期（结构与范围）+ 运行期（引用与状态合法性）。
10. ID 采用字符串主键（如 `shu_guanyu`），并附 `schema_version` 支持迁移。

### States and Transitions

- `StaticConfig` 生命周期：`Draft -> Validated -> Loaded -> Active`
- `RuntimeInstance` 生命周期：`Pool -> ShopOffer -> OwnedBench -> Deployed -> InBattle -> (Survived | Dead) -> (BenchReset | Recycled)`
- `SynergyRuntime` 生命周期：`Inactive -> Candidate -> Active(tier_n) -> Inactive`
- 合法转移约束：
	- `ShopOffer -> Deployed` 不允许（必须先 `OwnedBench`）
	- `Dead -> Deployed` 不允许（需通过复活机制，MVP 不支持）
	- `Active(tier_n) -> Active(tier_m)` 仅在单位变更后触发

### Interactions with Other Systems

- 对 `对局状态机`：在 `SHOP/DEPLOY/BATTLE/RESOLVE` 阶段提供只读数据视图。
- 对 `招募商店与刷新经济`：提供可抽取武将池与成本字段；接收购买/出售事件更新实例归属。
- 对 `九宫格布阵规则`：提供 `board_pos` 与单位可部署性校验。
- 对 `自动战斗结算`：提供战斗输入快照（属性、标签、技能引用、羁绊层级）。
- 对 `羁绊判定与加成`：提供标签与阈值定义；接收回写的当前激活层级（实例态）。
- 对 `战斗HUD与商店UI`：提供展示字段（名称、阵营、职业、星级、激活羁绊数）。

## Formulas

### 星级倍率

$$
M_{star}(1)=1.0,\;M_{star}(2)=1.8,\;M_{star}(3)=3.0
$$

### 实例基础属性映射

$$
Stat_{runtime}=Stat_{base}\cdot M_{star}\cdot M_{level}
$$

其中 $M_{level}=1+0.05\cdot(level-1)$，MVP 默认 `level=1`。

### 羁绊激活判定

$$
count(s)=\sum_{u\in Deployed,alive} \mathbf{1}[tag_s \in tags(u)]
$$

### 羁绊层级

$$
tier(s)=\max\{k\mid threshold_{s,k}\le count(s)\}
$$

### 单位战力评分（供商店/波次参考，不直接用于伤害）

$$
Power(u)=a\cdot ATK+b\cdot HP+c\cdot DEF+d\cdot SPD+e\cdot SkillWeight
$$

MVP 默认系数：$a=1.0,b=0.25,c=0.6,d=0.4,e=8.0$。

### 数据完整性校验

$$
valid\_ref = [general.skill\_id \in skill\_table]\land[synergy.tag \in tag\_registry]
$$

不通过则加载失败并阻止开局。

## Edge Cases

1. `general_id` 重复：加载即失败，输出冲突清单。
2. 武将引用不存在羁绊标签：加载警告并剔除非法标签。
3. 羁绊阈值无序（如 `[4,2,6]`）：自动排序并记录修正日志。
4. 阈值大于可上阵上限：标记为不可达层级，UI 不展示。
5. 购买同名武将导致升星时，实例合并需保留可追溯来源 ID 列表。
6. 单位死亡后仍被计入羁绊：禁止，羁绊计算仅统计 `alive && deployed`。
7. 同单位多标签触发多羁绊冲突：按 `priority` 决定先后，叠加规则由羁绊系统执行。
8. `board_pos` 越界或重复占位：部署校验失败，回退到候场。
9. 数据热更新后旧存档字段缺失：通过 `schema_version` 迁移补默认值。
10. 浮点误差导致阈值抖动：羁绊计数全用整数，禁止浮点比较阈值。

## Dependencies

### 上游依赖

- 无硬上游（Foundation）。
- 软依赖：`平衡参数配置表`（属性与羁绊效果参数来源）、`RNG与随机种子管理`（商店抽取与实例分配顺序可复现）。

### 下游依赖（关键接口）

1. `九宫格布阵规则`：读取 `RuntimeInstance.board_pos / deployable`。
2. `自动战斗结算`：读取战斗快照（属性、标签、星级、技能引用）。
3. `招募商店与刷新经济`：读取武将池静态表与 `cost`，写入购买后的实例归属。
4. `羁绊判定与加成`：读取标签与阈值，回写 `active_synergy_tiers`。
5. `战斗HUD与商店UI`：读取展示字段与激活羁绊。
6. `存档/读档（基础）`：读取/写入局内实例与 `schema_version`。

### 接口契约

- 静态表只读；实例表仅允许通过统一命令通道写入（`buy/sell/deploy/merge/resolve`）。
- 所有写操作返回 `revision`（递增版本）供 UI 与日志做一致性校验。
- 非法引用（不存在 ID）一律拒绝并记录错误，不进行隐式兜底。

## Tuning Knobs

| Knob | 默认值 | 安全范围 | 说明 |
|---|---:|---:|---|
| `max_star_level` | 3 | 2–4 | 控制升星上限 |
| `star_merge_count` | 3 | 2–4 | 几合一升星 |
| `tag_slots_per_general` | 2 | 1–3 | 每武将标签槽数 |
| `max_synergy_tiers` | 3 | 2–4 | 单羁绊层级数 |
| `base_stat_scale_global` | 1.0 | 0.8–1.2 | 全局属性缩放 |
| `cost_pool_weights` | [55,30,12,3] | 总和=100 | 1~4费出现权重 |
| `schema_strict_mode` | true | bool | 严格校验开关 |
| `invalid_tag_policy` | strip | strip/error | 非法标签处理策略 |
| `runtime_revision_limit` | 99999 | 1000–1e6 | 单局修订号上限 |
| `power_score_coeff` | a,b,c,d,e | 按公式范围 | 战力评分系数 |

联动约束：`max_star_level`、`star_merge_count` 与商店出货权重强耦合，需联调。

## Visual/Audio Requirements

- 羁绊激活层级变化需要统一事件（`synergy_tier_changed`）供特效与音效系统消费。
- 升星合成需要输出标准化反馈字段：`from_star`, `to_star`, `consumed_instance_ids`。
- 数据模型本身不负责表现，但必须保证事件字段稳定且可回放。

## UI Requirements

- 武将卡面最少显示：名称、阵营标签、职业标签、费用、星级。
- 羁绊面板最少显示：当前计数/阈值、当前层级、下一层所需数量。
- 发生非法数据（缺失引用/越界）时，调试 UI 可展示具体字段路径。
- UI 读取使用只读快照，禁止直接改写实例数据。

## Acceptance Criteria

1. 成功加载 12–18 名武将、6–8 条羁绊，且无重复 `general_id/synergy_id`。
2. 所有武将 `skill_id/tag` 引用可解析，非法引用会阻止开局并给出错误报告。
3. 购买/出售/部署/升星后，`RuntimeInstance.revision` 严格递增且无跳回。
4. 升星规则满足 `3合1`，升星后属性按公式更新且可复算一致。
5. 羁绊计数仅基于 `alive && deployed` 单位，候场位不计入。
6. 在 1000 次随机构筑模拟中，羁绊层级判定无越界、无负值、无抖动。
7. 数据模型快照可被战斗系统消费，字段缺失率为 `0`。
8. 对局状态切换时（`SHOP/DEPLOY/BATTLE`）读取一致快照，无跨状态脏读。
9. 旧版本存档（`schema_version` 落后 1 版）可迁移成功并通过校验。
10. 安卓目标设备上，单次数据重算耗时 `< 2ms`（MVP 规模）。
