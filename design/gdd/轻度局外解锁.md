# 轻度局外解锁

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 5分钟一局、低门槛高上限

## Overview

轻度局外解锁用于把“短局结果”转换成“可持续目标”，在不破坏单局公平性的前提下提供长期留存动力。系统仅解锁轻量内容（阵容槽位外观、初始展示位、可选起始提示、图鉴条目等），避免数值型永久成长导致“越玩越碾压”。

## Player Fantasy

玩家应感受到“每局都有积累，但不会因为局外成长把战斗变成无脑碾压”。解锁节奏应短、明确、可预期：打一两局就能看到一项新内容，同时新手不会因未解锁而被核心体验拒之门外。

## Detailed Design

### Core Rules

1. 局外解锁只消费“局外点数 `meta_point`”，`meta_point` 由局内结算统一发放。
2. `meta_point` 发放时机仅限每局终局（`GAME_WIN/GAME_OVER`），中途不入账。
3. 仅允许“功能解锁/体验增强”类项目；禁止直接提高战斗数值（攻击、生命、羁绊倍率）。
4. 每个解锁项包含：`unlock_id/type/cost/prereq_ids/max_level`。
5. 解锁流程必须原子化：校验前置 -> 扣点 -> 写入解锁状态 -> 记录日志。
6. 同一 `unlock_id` 重复请求需幂等：已解锁返回成功且不重复扣点。
7. 局外解锁状态持久化依赖 `存档/读档（基础）` 的稳定快照能力。
8. 所有解锁操作写入 `Playtest事件日志`，含前后点数与失败原因。
9. MVP 仅开放 6–10 个解锁项，控制复杂度与调参面。
10. 若检测到数据版本变更，支持按 `unlock_schema_version` 做迁移或安全回退。

### States and Transitions

- `Locked`：未解锁。
- `Unlockable`：前置满足且点数足够。
- `Unlocking`：事务执行中。
- `Unlocked`：已解锁。
- `Invalid`：配置异常或迁移失败，禁止操作。

合法转移：`Locked -> Unlockable -> Unlocking -> Unlocked`；异常分支：`Unlocking -> Locked`（事务回滚）、`Locked/Unlockable -> Invalid`（配置损坏）。

### Interactions with Other Systems

- `局内资源与奖励流`：在终局输出 `meta_point_delta`，作为局外货币来源。
- `存档/读档（基础）`：持久化 `meta_progress` 与 `unlock_state`。
- `对局状态机`：终局触发结算并切换至局外界面。
- `战斗HUD与商店UI`：展示“本局获得局外点数”和可解锁提示。
- `Playtest事件日志`：记录每次 unlock 请求、结果与耗时。

## Formulas

### 终局点数发放

$$
MetaPointGain = Base_{mode} + WinBonus + WaveBonus + StreakBonus
$$

MVP 默认：`Base=10`，`WinBonus=+5`（胜利），`WaveBonus=wave_cleared`，`StreakBonus` 默认关闭。

### 可解锁判定

$$
Unlockable = \mathbf{1}[meta\_point \ge cost] \land \mathbf{1}[prereq\_satisfied] \land \mathbf{1}[not\_unlocked]
$$

### 扣点后余额

$$
meta\_point' = clamp(meta\_point - cost,\ 0,\ meta\_cap)
$$

### 解锁进度完成率

$$
Completion = \frac{|UnlockedSet|}{|UnlockCatalog|}
$$

### 幂等事务键

$$
TxnKey = hash(player\_id, unlock\_id, request\_seq)
$$

## Edge Cases

1. 终局重复结算导致点数重复发放：使用 `match_id + reward_revision` 幂等去重。
2. 扣点成功但写状态失败：事务回滚并恢复点数。
3. 前置已满足但配置缺失：标记 `unlock_config_missing`，该项临时不可用。
4. 离线闪退发生在解锁中：下次启动按事务日志重放或回滚。
5. 迁移后出现非法等级（超 `max_level`）：裁剪并记录 `unlock_level_clamped`。
6. 点数上限溢出：统一 clamp 到 `meta_cap`。
7. 旧版本客户端请求新解锁项：返回 `schema_incompatible`。
8. 已解锁项再次点击：返回成功提示但不扣点。
9. 并发点击多个解锁：串行队列处理，避免竞争扣点。
10. 作弊篡改本地存档：校验哈希失败时回退最近有效快照。

## Dependencies

### 上游依赖（Hard）

- `存档/读档（基础）`：提供可靠持久化与恢复。
- `局内资源与奖励流`：提供终局点数来源与修订号。

### 下游依赖

1. `新手引导与失败复盘`（Soft）：可根据已解锁项调整引导文案。
2. `设置与可访问性增强`（Soft）：可把部分可访问性开关纳入局外解锁展示（非战斗数值）。

### 接口契约

- 输入最小字段：`player_id,unlock_id,meta_point,prereq_state,unlock_schema_version`。
- 输出最小字段：`unlock_result,meta_point_after,unlock_revision,error_code`。
- 约束：`unlock_revision` 必须单调递增；同 `TxnKey` 仅可应用一次。

## Tuning Knobs

| Knob | 默认值 | 范围 | 说明 |
|---|---:|---:|---|
| `meta_point_base` | 10 | 5–30 | 每局基础点数 |
| `meta_point_win_bonus` | 5 | 0–20 | 胜利额外点数 |
| `meta_point_wave_bonus` | 1/波 | 0–3/波 | 通关波次奖励 |
| `meta_cap` | 999 | 100–9999 | 局外点数上限 |
| `unlock_items_mvp_count` | 8 | 6–12 | MVP 解锁项数量 |
| `unlock_cost_curve` | linear | linear/step | 成本曲线 |
| `unlock_txn_timeout_ms` | 100 | 20–500 | 解锁事务超时 |
| `strict_prereq_check` | true | bool | 是否严格检查前置 |
| `enable_streak_bonus` | false | bool | 是否开启连胜额外点数 |
| `unlock_schema_version` | 1 | 1–99 | 解锁数据版本 |

## Visual/Audio Requirements

- 终局页显示“本局获得局外点数 + 当前余额”。
- 解锁成功使用轻量正反馈（短闪 + 提示音）；失败仅提示原因，不做强打断。
- 新解锁项首次获得时可显示一次“NEW”标记（<2s）。

## UI Requirements

- 局外“解锁面板”仅包含：当前点数、解锁树列表、单项详情、解锁按钮。
- 每个解锁项展示：名称、效果说明、花费、前置条件、当前状态。
- 不展示复杂分页与筛选（MVP 简化）。
- 终局结算页提供“前往解锁”快捷入口。

## Acceptance Criteria

1. 终局发放点数与战斗结果一致，重放 1000 次无重复发放。
2. 所有解锁项均不直接修改战斗核心数值。
3. 解锁事务具备幂等与回滚能力，断电/闪退后数据一致。
4. 已解锁项重复点击不扣点，返回幂等成功。
5. 前置条件与花费校验准确，非法请求被拒绝并可追踪。
6. 解锁数据可被 `存档/读档（基础）` 正确保存与恢复。
7. 玩家在 2–3 局内至少完成 1 次解锁（默认参数下）。
8. UI 信息清晰，新手可在 30 秒内理解“点数-解锁”关系。
9. Playtest 日志可完整追踪点数变化与解锁事件。
10. MVP 复杂度受控（解锁项数量、操作路径、异常分支均可测试）。
