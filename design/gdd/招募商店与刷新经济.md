# 招募商店与刷新经济

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 羁绊优先、5分钟一局、低门槛高上限

## Overview

招募商店与刷新经济负责在 `SHOP` 阶段提供可购买武将、刷新决策与金币流转规则，决定玩家在短局内如何完成阵容构筑与转型。系统目标是在 5 分钟单局里做到“可规划、可博弈、可复盘”：玩家既能通过经济管理获得优势，也不会因数值过重导致新手难以上手。

## Player Fantasy

玩家应感受到“我靠运营把阵容凑出来了”。我可以选择即时提强度保当前波，也可以攒钱赌下一波高价值转型；每次刷新都不是纯运气，而是围绕羁绊方向和经济节奏的有意义决策。

## Detailed Design

### Core Rules

1. 仅在状态机 `SHOP` 阶段开放商店操作；其他阶段拒绝买卖与刷新请求。
2. 每次商店展示 `shop_slot_count` 个武将（MVP 默认 5 格）。
3. 玩家可执行：购买、出售、手动刷新、锁定商店。
4. 购买成功条件：金币足够 + 候场未满 + 目标槽位仍有效。
5. 出售返还固定比例金币（默认 `sell_refund_ratio=1.0`，即按原价返还，MVP 友好）。
6. 每波开始会按规则自动刷新商店；若上波锁定则保留原列表并清除锁定。
7. 手动刷新消耗固定金币（默认 1），刷新后生成新候选列表。
8. 商店出货概率按费用分层权重决定，并受当前波次影响（非玩家等级制，适配短局）。
9. 重复购买同名武将时，数据模型侧按 `3合1` 升星规则处理；商店仅负责交易与库存。
10. 所有经济操作必须写事件日志：`before_gold`、`delta_gold`、`after_gold`、`op_type`。

### States and Transitions

- `Closed`：非 `SHOP` 阶段，拒绝全部商店写操作。
- `OpenUnlocked`：可买卖、可刷新、未锁定。
- `OpenLocked`：本波结束时保留当前商店内容。
- `Settling`：阶段结束，生成关闭事件并回传状态机。

合法转移：`Closed -> OpenUnlocked -> (OpenLocked | OpenUnlocked) -> Settling -> Closed`。

### Interactions with Other Systems

- `对局状态机`：接收 `state_enter(SHOP)` 打开商店；输出 `shop_ops_closed`。
- `武将/羁绊数据模型`：读取武将池与费用，写入购买/出售后的实例归属。
- `RNG与随机种子管理`：提供可复现抽样序列与刷新随机源。
- `平衡参数配置表`：提供费率权重、刷新成本、利息上限等参数。
- `九宫格布阵规则`：消费购买结果（候场新增单位）。
- `局内资源与奖励流`：在波后结算发放金币，并反馈到下个 `SHOP`。
- `战斗HUD与商店UI`：展示商店格子、金币变化、锁店状态、操作禁用态。
- `Playtest事件日志`：记录刷新次数、花费结构、回合末剩余金币分布。

## Formulas

### 波次初始金币

$$
GoldStart_w = GoldBase_w + WinBonus_w + StreakBonus_w
$$

MVP 默认：$GoldBase=\{6,7,8\}$（对应 Wave1/2/3）。

### 刷新成本

$$
RefreshCost = C_{refresh}
$$

MVP 默认 $C_{refresh}=1$。

### 利息收入（回合末）

$$
Interest = \min(I_{cap},\ \lfloor Gold_{end}/G_{step}\rfloor\cdot I_{step})
$$

MVP 默认：$G_{step}=10,\ I_{step}=1,\ I_{cap}=3$。

### 出货概率（按费用层）

$$
P(cost=k\mid wave=w)=W_{w,k}/\sum_j W_{w,j}
$$

MVP 建议权重（1~4费）：
- Wave1: `[60,30,10,0]`
- Wave2: `[45,35,17,3]`
- Wave3: `[30,35,25,10]`

### 购买与出售后的金币

$$
Gold' = Gold - Cost(unit)
$$

$$
Gold'' = Gold + \lfloor Cost(unit)\cdot sell\_refund\_ratio \rfloor
$$

### 保底机制（可选开关）

$$
if\ refresh\_without\_target\_tag \ge N \Rightarrow force\_one\_slot\_from\_target\_pool
$$

MVP 默认关闭，作为平衡开关。

## Edge Cases

1. 金币不足仍购买/刷新：拒绝操作并提示 `insufficient_gold`。
2. 候场已满仍购买：拒绝购买并提示 `bench_full`。
3. 同一槽位被重复点击购买：仅首个请求生效，其余按幂等键忽略。
4. 锁店状态下进入新波：保留一次后自动解锁，防止永久锁定。
5. 商店池耗尽（某费段无可抽单位）：回退到相邻费段或全池重采样，并记录告警。
6. 切后台恢复导致重复扣费：按事务ID校验，重复请求不重复扣金币。
7. 刷新与阶段结束同帧：以状态机关店优先，刷新请求返回 `shop_closed`。
8. 出售单位恰好触发升星回退：以数据模型最终 `revision` 为准回写 UI。
9. 非法武将ID进入商店：该槽位作废并立即补抽，记录 `invalid_offer_recovered`。
10. 负金币/超上限金币：统一 clamp 到 `[0, gold_cap]`。

## Dependencies

### 上游依赖（Hard）

- `对局状态机`：提供 SHOP 窗口生命周期。
- `武将/羁绊数据模型`：提供可抽取武将池与交易写入。
- `RNG与随机种子管理`：保证刷新可复现。
- `平衡参数配置表`：提供概率、价格、利息等参数。

### 下游依赖

1. `九宫格布阵规则`（Hard）：消费候场新增单位与可部署状态。
2. `局内资源与奖励流`（Hard）：消费本波经济快照并回写奖励。
3. `战斗HUD与商店UI`（Hard）：消费商店数据与操作反馈。
4. `Playtest事件日志`（Soft）：消费运营行为统计。

### 接口契约

- 输入最小字段：`match_id,wave,gold,shop_state,seed,pool_revision`。
- 输出最小字段：`offers[],gold_after,shop_locked,transaction_id`。
- 所有经济写操作必须返回递增 `economy_revision`。

## Tuning Knobs

| Knob | 默认值 | 范围 | 说明 |
|---|---:|---:|---|
| `shop_slot_count` | 5 | 4–7 | 商店展示槽位数 |
| `refresh_cost` | 1 | 0–3 | 单次刷新消耗 |
| `sell_refund_ratio` | 1.0 | 0.5–1.0 | 出售返还比例 |
| `gold_base_by_wave` | [6,7,8] | 每项4–12 | 波次基础金币 |
| `interest_step_gold` | 10 | 5–20 | 利息档位金币间隔 |
| `interest_step_value` | 1 | 0–2 | 每档利息 |
| `interest_cap` | 3 | 0–5 | 利息上限 |
| `shop_weights_wave1` | [60,30,10,0] | 总和100 | W1费率权重 |
| `shop_weights_wave2` | [45,35,17,3] | 总和100 | W2费率权重 |
| `shop_weights_wave3` | [30,35,25,10] | 总和100 | W3费率权重 |

| `pity_refresh_n` | 0 | 0–8 | 保底刷新阈值（0=关闭） |

## Visual/Audio Requirements

- 购买成功：金币扣减飘字 + 槽位淡出补位（<200ms）。
- 刷新成功：整行轻微翻牌过渡，避免强动画拖慢节奏。
- 锁店状态：明确锁图标与提示色，不与普通状态混淆。
- 金币不足/候场满：统一失败音与红色提示文案。

## UI Requirements

- `SHOP` 顶栏显示：当前金币、预计利息、刷新成本、锁店状态。
- 每个商店槽位显示：武将头像、费用、阵营/职业标签、星级潜力提示（可选）。
- 提供一键刷新按钮与锁店按钮，非 `SHOP` 阶段置灰不可点。
- 购买后若可触发升星，显示非阻塞提示（不弹窗打断）。
- 调试面板可查看当前波次费率权重与最近5次刷新结果。

## Acceptance Criteria

1. `SHOP` 阶段内买/卖/刷/锁全流程可用，成功与失败反馈明确。
2. 非 `SHOP` 阶段所有商店写请求被拒绝且有日志。
3. 1000 次同种子模拟下，刷新结果分布与理论权重误差在可接受范围内（±3%）。
4. 任意交易后金币不出现负值，`economy_revision` 严格递增。
5. 锁店逻辑稳定：可保留一次并在下一波自动解锁。
6. 候场满、金币不足、关店边界等场景不出现卡死或重复扣费。
7. 购买结果可被数据模型正确消费并进入候场。
8. 平均单次刷新逻辑耗时 `< 1ms`（MVP 规模）。
9. 玩家在 3 局内能理解“刷新-攒钱-利息”的基本运营关系。
10. 运营行为日志可支持复盘“何时花钱、为何转型”的分析。
