# 存档/读档（基础）

> **Status**: Approved
> **Author**: 用户 + GitHub Copilot
> **Last Updated**: 2026-03-25
> **Implements Pillar**: 5分钟一局、低门槛高上限

## Overview

存档/读档（基础）负责在关键状态点持久化对局快照，并在恢复时重建“可继续、可复现、可追溯”的运行上下文。系统目标是解决移动端切后台、闪退、短时中断等场景下的进度丢失问题，同时不破坏状态机幂等性与 RNG 可复现性。

## Player Fantasy

玩家应感受到“我随时可以继续这局，而不是被迫重开”。恢复后看到的资源、波次、阵容与商店状态都与中断前一致；失败或胜利仍然可解释，不会出现“恢复后变随机”或“奖励重复发放”。

## Detailed Design

### Core Rules

1. 仅允许在状态机定义的安全点写快照：`WAVE_PREPARE` 进入后、`SHOP` 结束后、`RESOLVE` 完成后。
2. 快照必须是原子写入：先写临时文件，再校验，再替换正式快照。
3. 快照最小集合必须包含：状态机状态、波次索引、资源快照、阵容快照、RNG游标、版本号。
4. 读档仅允许从“最近一次有效快照”恢复，校验失败时禁止继续并回退重开。
5. 单局绑定固定 `match_id` 与 `balance_data_version`，恢复后不得漂移。
6. 同一快照不得重复应用；需用 `snapshot_id` + `resource_revision` 做幂等防重。
7. 快照写入不阻塞主循环：使用异步写队列，超时则降级记录告警。
8. 读档成功后必须触发一次全量一致性重算（资源、羁绊、UI缓存）。
9. 发布构建只保留单槽自动存档（MVP），调试构建可多槽。
10. 终局状态（`GAME_WIN/GAME_OVER`）读档后只能进入结果页，不可回到战斗中途。

### States and Transitions

- `NoSnapshot`：当前无可用快照。
- `Saving`：快照采集与序列化中。
- `Saved`：存在可恢复快照。
- `Loading`：读取与校验快照中。
- `Recovered`：恢复完成，进入状态机重入流程。
- `Invalid`：快照损坏/版本不兼容，拒绝恢复。

合法转移：`NoSnapshot -> Saving -> Saved -> Loading -> Recovered`，异常分支：`Loading -> Invalid -> NoSnapshot`。

### Interactions with Other Systems

- `对局状态机`：提供安全快照时机、恢复后的重入状态。
- `局内资源与奖励流`：提供 `post_gold/post_hp/resource_revision` 关键恢复字段。
- `RNG与随机种子管理`：提供各随机流游标与 `rng_schema_version`。
- `九宫格布阵规则`：提供布阵快照与 `layout_revision`。
- `招募商店与刷新经济`：提供商店锁定状态、候选列表与 `economy_revision`。
- `羁绊判定与加成`：恢复后触发一次重算，保证层级一致。
- `战斗HUD与商店UI`：恢复后拉取统一快照并刷新显示。
- `Playtest事件日志`：记录存读档操作、耗时、失败原因。

## Formulas

### 快照完整性校验

$$
Valid = \mathbf{1}[Hash(payload)=checksum_{stored}] \land \mathbf{1}[schema_{runtime}=schema_{snapshot}]
$$

### 恢复目标快照选择

$$
Snapshot_{use} = \arg\max_{s \in S_{valid}}(timestamp_s)
$$

### 幂等恢复判定

$$
Apply = \mathbf{1}[(snapshot\_id,resource\_revision) \notin AppliedSet]
$$

### 写入耗时约束

$$
SaveLatency = t_{fsync}-t_{enqueue} \le L_{max}
$$

MVP 建议 `L_max = 30ms`（异步路径）。

### 关键状态一致性

$$
Consistent = \mathbf{1}[state\_phase_{snapshot}=state\_phase_{restored}] \land \mathbf{1}[wave_{snapshot}=wave_{restored}]
$$

### RNG恢复一致性

$$
RngMatch = \mathbf{1}[cursor_{restored}=cursor_{snapshot}] \land \mathbf{1}[rng\_schema\_version\_match]
$$

## Edge Cases

1. 快照文件半写入（断电/闪退）：读取时校验失败，自动回退上一有效快照。
2. 快照版本过旧：尝试迁移；迁移失败则标记 `snapshot_incompatible`。
3. 读档时资源修订号倒退：拒绝应用并记录高优先级日志。
4. 战斗中途崩溃恢复：回退到最近 `SHOP/RESOLVE` 安全点，不回到未完成战斗 tick。
5. RNG 游标缺失：恢复失败并提示不可精确续局。
6. 商店候选列表丢失：按同种子 + 同游标重构；失败则重新生成并标记非精确恢复。
7. 终局后重复读档：仅允许查看结果，不允许重入对局。
8. 磁盘空间不足：写入失败但不崩溃，提示用户并继续本局内存态。
9. 多端时间戳异常：按单调递增 `snapshot_seq` 优先，不依赖系统时间。
10. 快速连续写入请求：合并队列，仅保留最后一次有效快照。

## Dependencies

### 上游依赖（Hard）

- `对局状态机`：提供可写快照时机与恢复重入入口。
- `局内资源与奖励流`：提供恢复必需的资源一致性字段。

### 下游依赖

1. `轻度局外解锁`（Hard）：消费稳定持久化能力保存局外进度。
2. `Playtest事件日志`（Soft）：消费存读档失败与恢复耗时数据。
3. `战斗HUD与商店UI`（Hard）：消费恢复结果刷新界面。

### 接口契约

- 输入最小字段：`match_id,snapshot_reason,state_phase,wave,resource_snapshot,rng_snapshot`。
- 输出最小字段：`snapshot_id,save_result,load_result,restore_state`。
- 约束：同一 `snapshot_id` 不可重复应用；恢复成功后必须回传 `restore_revision`。

## Tuning Knobs

| Knob | 默认值 | 范围 | 说明 |
|---|---:|---:|---|
| `autosave_enabled` | true | bool | 是否启用自动存档 |
| `autosave_max_slots` | 1(prod)/3(dev) | 1–10 | 自动存档槽位数 |
| `autosave_on_shop_exit` | true | bool | 退出商店是否存档 |
| `autosave_on_resolve` | true | bool | 结算后是否存档 |
| `snapshot_compress` | true | bool | 快照是否压缩 |
| `snapshot_encrypt` | false(MVP) | bool | 快照是否加密 |
| `save_latency_budget_ms` | 30 | 10–100 | 写入延迟预算 |
| `load_timeout_ms` | 200 | 50–1000 | 读档超时预算 |
| `snapshot_keep_days` | 3 | 1–30 | 快照保留天数 |
| `strict_restore_mode` | true | bool | 严格一致恢复开关 |

## Visual/Audio Requirements

- 存档成功采用轻提示（不打断流程，<1.2s）。
- 读档恢复时显示短加载遮罩与“继续对局”提示。
- 读档失败提供明确失败原因文案（版本不兼容/文件损坏/空间不足）。

## UI Requirements

- 主界面显示“继续上局”入口（仅有有效快照时显示）。
- 对局中不显示手动存档按钮（MVP），仅自动存档提示。
- 调试构建提供“导出快照/导入快照/强制损坏测试”入口。
- 恢复后 HUD 必须显示当前波次、金币、生命与快照前一致。

## Acceptance Criteria

1. 切后台/杀进程后可从最近有效快照恢复，成功率 `>=99%`（测试样本）。
2. 恢复后状态机阶段、波次、资源与阵容一致，无越权跳状态。
3. 同快照重复加载不会导致奖励重复或资源翻倍。
4. RNG 游标恢复后随机序列连续一致。
5. 快照损坏/不兼容时可安全降级，不崩溃。
6. 自动存档平均耗时满足 `save_latency_budget_ms`。
7. 读档失败路径提示清晰，且可回到主菜单重开。
8. 终局读档不可回滚到战斗中途。
9. 日志可完整追踪每次 save/load 的结果与原因。
10. 为后续 `轻度局外解锁` 提供稳定持久化基座。
